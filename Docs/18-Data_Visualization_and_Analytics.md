# 📊 محرك البيانات والتحليلات (Data Visualization & Analytics Engine)

تعتمد Noble Architecture على نظام متطور لعرض البيانات وتوليد التقارير الحية التي تساعد صناع القرار في مراقبة أداء المؤسسة.

## 1. التكنولوجيا المستخدمة
يستخدم النظام مزيجاً من الأدوات لضمان السرعة والجمالية:
- **Frontend:** مكتبة `Recharts` و `Lucide Icons` لعرض الرسوم البيانية المتجاورة والمستجيبة.
- **Backend:** محركات استعلام متقدمة في Eloquent تقوم بمعالجة البيانات وتجميعها (Aggregation) قبل إرسالها للواجهة.
- **Theme Sync:** جميع المخططات البيانية تتغير ألوانها تلقائياً عند التبديل بين الوضع اللليلي والنهاري بناءً على Geist Tokens.

## 2. أنواع المخططات المدعومة
| المخطط | الاستخدام | المكون (Component) |
|--------|-----------|--------------------|
| **Area Chart** | مراقبة التدفقات النقدية والنمو الزمني. | `AreaChart.tsx` |
| **Bar Chart** | المقارنة بين الأقسام أو الموظفين. | `BarChart.tsx` |
| **Pie Chart** | توزيع الموارد أو فئات الإيرادات. | `PieChart.tsx` |
| **Radar Chart** | تقييم أداء الموظفين في HRM. | `RadarChart.tsx` |

## 3. هيكلية البيانات (Data Shape)
يتوقع النظام وصول البيانات بتنسيق JSON موحد لسهولة الرندرة:
```json
[
  { "month": "Jan", "revenue": 4500, "expenses": 3200 },
  { "month": "Feb", "revenue": 5200, "expenses": 3100 }
]
```

## 4. نصائح للأداء (Performance Tips)
1. **Lazy Loading:** يتم تحميل المخططات البيانية الثقيلة فقط عند ظهورها في مجال رؤية المستخدم (Viewport).
2. **Memoization:** نستخدم `React.memo` لمنع إعادة رسم المخططات (Re-rendering) ما لم تتغير البيانات الفعلية.
3. **Pre-aggregation:** يُنصح دائماً بتجميع البيانات في السيرفر (SQL level) بدلاً من إرسال آلاف السجلات ليقوم المتصفح بمعالجتها.

## 5. إضافة مقياس جديد (New Metric)
لإضافة رسم بياني جديد، يجب إنشاء دالة في الـ Controller الخاص بالموديول تعيد البيانات، ثم استدعاء مكون المخطط المناسب داخل صفحة الـ Dashboard وتمرير البيانات إليه كـ `props`.
